Générateur de dessins ASCII en Scala

Contexte

La société FunProg veut développer en Scala un générateur de dessin utilisable dans le terminal, sans interface graphique. Le programme devra lire une suite de commandes textuelles, les analyser, les interpréter, puis afficher un dessin ASCII dans le terminal.

Notions abordées et consignes

L’objectif de ce projet sera de concevoir et écrire un programme Scala implémentant la spécification ci-après tout en mettant en pratique les notions fondamentales de la programmation fonctionnelle abordés: - pattern matching - récursivité - fonction ordre supérieur - manipulation de listes - immutabilité - types algébriques (sealed trait, case class) - séparation entre logique pure et entrées/sorties

De plus le dit porgramme devra respecter les consignes suivantes (Zéro sera donné en cas de non-respect de ses consignes):

  • il est INTERDIT d’utiliser le mot clé return
  • il est INTERDIT d’utiliser le mot-clé while
  • il est INTERDIT d’utiliser le mot-clé null
  • il est INTERDIT d’utiliser les expressions régulières
  • tout if devra être exaustif (c-à-d avec un else)
  • tout pattern-matching devra être exaustif (avec un cas par défaut sinon, ou sans warning si pas besoin de cas par défaut - avec collect par exemple - )
  • la mutabilité explicite (var) ou implicite (struture de données mutables du package scala.collection.mutable comme ArrayBuffer) est INTERDITE
  • il sera INTERDIT de throw des exceptions, privilégier Try avec exit du programme en cas d’erreur fatale
  • seules les implémentations valides (qui compilent) seront prises en compte et seules celles renvoyant un résultat correct donneront lieu à la totalité des points

Un template de projet sbt sera fourni. Celui-ci servira de base pour les devéloppements. Il n’est pas permis de changer les options de compilateurs définis pour ce projet. De plus, il n’est pas permis d’importer des libs autres que celles fournies dans le template de projet sbt.

Détail du programme

  1. Objectif du projet

Notre générateur de dessin sera capable d’interpréter un petit langage de dessin comme ceci:

canvas 20 8
point 2 3
line 1 1 10 1
rect 5 2 6 3

et d’afficher un résultat du type :

....................
.##########.........
.....######.........
..*..#....#.........
.....######.........
....................
....................
....................

Le caractère . représente une case vide. Les autres caractères représentent les éléments dessinés.

  1. Fonctionnement attendu

Le programme fonctionne dans le terminal. Il lit des commandes, construit une représentation interne du dessin, puis affiche la grille finale.

Deux modes sont possibles selon votre choix : - mode interactif : l’utilisateur saisit les commandes une par une ; - mode script : l’utilisateur saisit plusieurs lignes, puis une commande render affiche le résultat.

Exemple :

AsciiDraw> canvas 10 5
AsciiDraw> point 1 1
AsciiDraw> line 2 2 8 2
AsciiDraw> render
..........
.*........
..#######.
..........
..........

Une commande comme quit ou exit permet de terminer le programme.

  1. Fonctionnalités à implémenter

Niveau 1 — Canevas et point

Le programme doit reconnaître : - la création d’un canevas : canvas largeur hauteur - le dessin d’un point : point x y

Exemples :

canvas 10 5
point 3 2

Niveau 2 — Lignes horizontales et verticales

Ajouter le dessin de lignes :

line x1 y1 x2 y2

Dans une première version, vous pouvez limiter line aux lignes horizontales et verticales.

Exemples :

line 1 1 6 1
line 4 1 4 4

Niveau 3 — Rectangles

Ajouter :

rect x y largeur hauteur

Le rectangle peut être dessiné uniquement avec son contour.

Exemple :

rect 2 1 5 3

Niveau 4 — Remplissage et effacement

Ajouter une ou deux commandes parmi les suivantes :

fill x y o
clear
setchar c

Interprétation possible : - fill x y o remplit avec le caractère o une zone connexe (point . différente de # ) - clear vide le canevas - setchar - change le caractère utilisé pour les dessins suivants en -

  1. Gestion d’erreurs

Lors du parsing des données d’entrées, si les données attendues ne sont pas fournies ou ne sont pas au format attendu, aucune exception ne devra être throw. Il est recommandé de créer une hiérarchie d’erreur propre à votre programme et vos traitements.

Le programme doit gérer proprement quelques erreurs simples : - commande inconnue - nombre d’arguments incorrect - coordonnées en dehors du canevas - tentative de dessiner avant la création d’un canevas - dimensions invalides

Exemples :

AsciiDraw> point 2 3
Erreur: aucun canevas n'a été créé

AsciiDraw> rect 1 2
Erreur: nombre d'arguments invalide pour rect

Voir Tryou Either pour gestion et propagation des erreurs. Veuillez aussi à bien limiter les effets de bord et gestions d’erreur aux frontières de votre programme (entrées/sorties) et non au coeur de celui-ci.

  1. Exemples de tests attendus

Votre programme doit au minimum réussir les cas suivants :

Entrée Résultat attendu
canvas 5 3 puis render grille vide 5x3
canvas 5 3 puis point 1 1 un caractère tracé à la position (1,1)
line 1 1 4 1 ligne horizontale
line 2 0 2 2 ligne verticale
rect 1 1 3 2 rectangle dessiné correctement
clear canevas vidé

Le programme avec les commandes avancées suivantes:

canvas 12 6
setchar x
line 11 1 11 4
setchar #
rect 2 1 6 4
fill 3 2 o
render

Devra produire le dessin suivant:

............
..######...x
..#oooo#...x
..#oooo#...x
..######...x
............

Bonus

Si le projet de base est terminé, vous pouvez ajouter une section bonus avec les deux fonctionnalités suivantes.

Historique des états

Le programme conserve l’historique des dessins successifs afin de permettre une annulation.

Exemple :

AsciiDraw> canvas 10 5
AsciiDraw> point 1 1
AsciiDraw> rect 3 1 4 2
AsciiDraw> undo

Après undo, le rectangle disparaît mais le point reste présent.
